# Система страниц кейсов

Этот документ — источник правды для внутренних страниц портфолио. Он описывает
структуру кейсов, медиаблоки, локализацию и контракты будущих функций. Текущая
реализация СмартРезерв остаётся на TSX; переход на MDX и два языка выполняется
отдельной задачей.

## Цели системы

- Один шаблон и один набор компонентов для всех кейсов.
- Семантическая структура текста, пригодная для индексации.
- Предсказуемые правила изображений на десктопе и мобильных устройствах.
- Независимые русская и английская версии текста.
- Общие или локализованные медиа без дублирования одинаковых файлов.
- Возможность добавить переход карточка → обложка и доступ по паролю, не
  переделывая контент страницы.

## Маршруты и языки

Целевая схема маршрутов:

    /ru/<slug>/
    /en/<slug>/

- Slug одинаковый в обоих языках: переключатель меняет только locale.
- Канонический адрес и metadata задаются отдельно для каждой версии.
- Страница публикует hreflang для ru, en и при необходимости x-default.
- Текущий адрес /yandex-eats-smartreserve/ сохраняется как совместимый адрес
  русской версии после внедрения локализованных маршрутов.
- Переключатель языка ведёт на тот же кейс и сохраняет позицию только когда это
  не мешает пониманию страницы.
- Маршрут отсутствующего перевода не генерируется. Список доступных языков
  хранится в данных кейса, поэтому переключатель не должен вести на 404.

Корневой html получает язык интерфейса сайта. Контейнер кейса дополнительно
получает корректный lang, если язык контента отличается от языка оболочки.

## Целевая структура контента

После подключения MDX:

    site-nextjs/
    ├── content/
    │   └── cases/
    │       └── <slug>/
    │           ├── ru.mdx
    │           └── en.mdx
    ├── app/
    │   └── [locale]/
    │       └── [slug]/
    │           └── page.tsx
    └── data/
        └── cases.ts

data/cases.ts хранит только общие данные и состояние публикации:

    {
      slug: "yandex-eats-smartreserve",
      tag: "b2b",
      availableLocales: ["ru", "en"],
      transitionId: "yandex-eats-smartreserve",
      access: { scope: "public" },
    }

Локализованный MDX хранит заголовок, описание, SEO-метаданные, текст и порядок
контентных блоков. Общие значения не копируются в оба файла без необходимости.

## Семантика страницы

- На странице ровно один h1.
- Самостоятельные части истории оформляются как section с h2.
- Обычный связующий текст не создаёт лишнюю секцию.
- Перечни оформляются как ul или ol, а не набором визуальных абзацев.
- Изображение с подписью оформляется как figure + figcaption.
- Значимое изображение получает локализованный alt; декоративное — пустой alt.
- Язык страницы, title и description соответствуют фактическому контенту.

## Сетка и интервалы

Текстовая колонка фиксирована на десктопе и получает поля 16 px на мобильных:

| Блок | Десктоп | Мобильный |
| --- | ---: | ---: |
| inline | 684 px | ширина экрана минус 32 px |
| wide | 1156 px | от края до края |
| full | ширина viewport | от края до края |

- Высота изображения всегда следует его исходному соотношению сторон.
- Основной вертикальный интервал на десктопе — 80 px.
- На мобильном основной интервал между блоками — 64 px.
- Первый текст после обложки использует 80 px на десктопе и 40 px на
  мобильном.
- На мобильном между блоком заголовка кейса и обложкой всегда остаётся 12 px.
- Два медиаблока, которые стоят подряд без текста между ними, разделяются
  интервалом 12 px на всех размерах экрана.
- Парный блок использует две колонки на десктопе и одну колонку на мобильном;
  вертикальный интервал между изображениями на мобильном — 12 px.
- Контейнер пары может быть inline, wide или full.

## Медиакомпоненты

Общие компоненты:

- CaseMedia — одно изображение ширины inline, wide или full;
- CaseMediaPair — два изображения в одном контейнере;
- OptimizedImage — AVIF/WebP, srcset, размеры и placeholder из manifest.

Подпись опциональна. Размер контейнера не должен задавать фиксированную высоту
изображению.

## Имена и хранение медиа

Исходники проекта:

    media-originals/images/<slug>/

Паттерн имени:

    <номер-блока>-<смысл>[-<вариант>].<расширение>

Одинаковый номер объединяет файлы одного блока:

    02-platform-hostess.png
    02-platform-manager.png

Для двух языков целевая структура такая:

    media-originals/images/<slug>/
    ├── shared/
    ├── ru/
    └── en/

- В shared лежат фотографии, схемы без текста и одинаковые для обоих языков
  изображения.
- В ru и en лежат скриншоты или иллюстрации с локализованным текстом.
- В первой версии существующие файлы непосредственно в <slug>/ считаются
  общими. Перенос в shared не обязателен до внедрения языковых маршрутов.
- Если локализованного медиа ещё нет, страница может использовать общий файл
  или исходную языковую версию. Это решение задаётся явно в контенте и не
  должно приводить к отсутствующему изображению.
- Текст страницы и SEO переводятся независимо от готовности локализованных
  скриншотов.

Пайплайн рекурсивно сохраняет структуру подпапок в media manifest. Файлы в
public/media/ и generated/media-manifest.json вручную не редактируются.

## Появление страницы

При прямой загрузке первый экран повторяет характер появления snakeview:

- заголовок и обложка проявляются с подъездом снизу на 18 px;
- easing — cubic-bezier(0.16, 1, 0.3, 1);
- длительность — 560 ms;
- обложка появляется после заголовка с коротким stagger;
- остальной длинный контент не получает повторяющиеся reveal-анимации;
- при prefers-reduced-motion анимация отключена.

Будущий переход с главной использует одинаковый transitionId у карточки и
обложки. После реализации shared-element перехода обычная entrance-анимация не
должна конкурировать с ним.

## Доступ по паролю

CaseAccessBoundary сохраняет будущий контракт:

- public — открытая страница;
- page — защищена вся страница;
- section — защищён отдельный блок.

Сейчас реализован только публичный режим. На статическом GitHub Pages
клиентский пароль не обеспечивает настоящую секретность: защищённый контент
нельзя включать в публичный HTML или bundle. Для реальной защиты потребуется
серверная проверка или отдельный сервис доступа.

## Как добавить новый кейс сейчас

1. Добавить карточке slug и стабильный transitionId в data/projects.ts.
2. Создать app/<slug>/page.tsx.
3. Использовать CaseAccessBoundary, CaseMedia и CaseMediaPair.
4. Положить исходники в media-originals/images/<slug>/.
5. Проверить единый паттерн имён и соответствие номеров составным блокам.
6. Добавить локализованные metadata и семантические заголовки.
7. Выполнить pnpm build.
8. Проверить прямую ссылку и переход с карточки в desktop/mobile.

После внедрения MDX пункты 2 и 6 заменятся созданием ru.mdx и en.mdx.

## Быстрый контекст для нового чата

Перед изменениями нужно прочитать корневой README.md, затем этот документ.
Рабочая реализация находится только в site-nextjs; site-html и старые планы не
являются источником правды для страниц кейсов.

Точки входа текущей реализации:

- app/yandex-eats-smartreserve/page.tsx — структура и контент первого кейса;
- components/CaseMedia.tsx — одиночные и парные медиаблоки;
- components/OptimizedImage.tsx — responsive-изображения из manifest;
- components/CaseAccessBoundary.tsx — контракт будущего доступа;
- data/projects.ts — карточка, slug и transitionId;
- styles.css, раздел Case study system — общая сетка, типографика и адаптив;
- media-originals/images/yandex-eats-smartreserve/ — исходники проекта;
- scripts/prepare-media.mjs — общий медиапайплайн;
- generated/media-manifest.json и public/media/ — генерируемые файлы, которые
  нельзя редактировать вручную.

Что уже работает:

- адрес /yandex-eats-smartreserve/ и переход с карточки главной;
- семантическая TSX-страница, локальные Inter и Inter Tight;
- inline, wide, full, подписи и парные медиаблоки;
- на мобильном пары перестраиваются в одну колонку;
- entrance-анимация прямой загрузки и reduced-motion;
- единый transitionId подготовлен для будущего shared-element перехода;
- публичный режим CaseAccessBoundary.

Что намеренно отложено и не должно быть реализовано попутно:

- новая навигация и переключатель языка;
- MDX и маршруты /ru/<slug>/, /en/<slug>/;
- shared-element переход карточка → обложка;
- защита страницы или секции паролем;
- локализованные версии всех медиа.

Любое новое правило страницы сначала проверяется как общее правило системы
кейсов. Если оно общее, его нужно добавить в этот документ и реализовать через
общий компонент или общий CSS, а не только в странице СмартРезерва.

## Проверка перед публикацией

- Production-сборка и статический export проходят без ошибок.
- На странице один h1, корректная последовательность h2.
- inline, wide, full и пары соответствуют сетке.
- На мобильном между заголовком и обложкой есть 12 px.
- Нет горизонтального переполнения.
- Все изображения имеют корректный размер, srcset, alt и при необходимости
  подпись.
- Первый экран не остаётся скрытым при ошибке JavaScript.
- prefers-reduced-motion отключает entrance-анимацию.
- Карточка и обложка имеют одинаковый transitionId.
- Каждый опубликованный locale имеет собственные title, description,
  canonical и корректные ссылки переключателя.
- Отсутствующее локализованное медиа использует явно заданный fallback.
